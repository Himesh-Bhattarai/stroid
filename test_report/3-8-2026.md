## Subscriber Benchmark
- Benchmark script: `scripts/subscriber-benchmark.ts`
- Environment:
  - Node `v22.14.0`
  - Windows `x64`
  - store shape: `{ value: number }`
  - history disabled with `historyLimit: 0`
  - latency measured end-to-end from `setStore()` call until the notification flush reached a final marker subscriber
- Thresholds used for interpretation:
  - noticeable slowdown: `5ms`
  - visible stall: `16ms`
  - impractical single update: `100ms`
  - significant retained/peak heap growth: `8MB`

### No-op Subscribers
- up to `200,000` subscribers:
  - single update median stayed under `5.5ms`
  - `100` sequential updates stayed under `475ms`
  - retained heap growth stayed below `0.1MB`
- higher-count targeted run:
  - around `250,000` subscribers, single update crossed `10ms` and `100` sequential updates crossed `1s`
  - `50ms` single-update latency was not reached at `500,000`
  - highest stable run completed at `500,000` subscribers

### Small-Compute Subscribers
- up to `200,000` subscribers:
  - single update median stayed under `5ms`
  - `100` sequential updates stayed under `853ms`
  - retained heap growth stayed below `0.1MB`
- higher-count targeted run:
  - around `250,000` subscribers, single update crossed `10ms` and `100` sequential updates crossed `1s`
  - `50ms` single-update latency was not reached at `500,000`
  - highest stable run completed at `500,000` subscribers

### Read
- performance is broadly linear with subscriber count; there was noise from JIT/GC, but no cliff-collapse pattern in the tested range
- no meaningful retained-memory growth was observed through `500,000` subscribers in repeated update cycles
- the first practical UI-stall territory starts around the `250k` to `400k` band, depending on callback cost and runtime jitter
