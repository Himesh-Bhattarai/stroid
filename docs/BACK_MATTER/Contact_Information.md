# Contact Information

This documentation set does not pretend to have a formal publisher support desk behind it.

So the useful contact information is project-oriented, not decorative.

## For Documentation Corrections

Use the repository workflow where this documentation lives.

That includes:

- opening an issue
- opening a documentation fix
- leaving review comments on the relevant Markdown files

Most likely files to reference:

- [docs](/docs)

## For Code or Behavior Questions

Reference the actual source and tests when reporting a concern:

- [src](/src)
- [tests](/tests)

This is important because many documentation disagreements are really behavior questions.

## For Documentation Status Questions

Treat this documentation as:

- grounded in the current codebase
- organized as the current official long-form documentation set
- likely to evolve as the library evolves

## Honest Note

If this becomes a formal public documentation release later, this page should be replaced with:

- official issue/reporting links
- maintainer contact path
- release-specific support instructions
- documentation contribution rules

Until then, the honest contact model is simple:

the repository is the contact surface.
