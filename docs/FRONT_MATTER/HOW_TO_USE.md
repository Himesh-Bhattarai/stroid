# How to Use This Book

This book is structured like a technical manual, not a landing page.

That means you should read it differently depending on why you are here.

## If you are evaluating Stroid

Read in this order:

1. [Preface](/docs/FRONT_MATTER/PREFACE.md)
2. [Introduction](/docs/FRONT_MATTER/INTRODUCTION.md)
3. [Core of Stroid - Introduction](/docs/BODY_MATTER/CORE_OF_STROID/INTRODUCTION.md)
4. [Core of Stroid - Real Use](/docs/BODY_MATTER/CORE_OF_STROID/REAL_USE.md)
5. [Opt-In Features - Introduction](/docs/BODY_MATTER/OPT_IN_FEATURES_OF_STROID/INTRODUCTION.md)

This gives you the package model first, before you get lost in API details.

## How the writing is organized

Each unit and chapter follows a stable pattern:

- opener
- learning objectives
- chapter outline
- numbered sections
- examples or tables
- summary
- review questions
- exercises
- references

That pattern is intentional.

It makes the documentation predictable, which reduces mental switching cost.

## How honesty is handled in this book

This documentation distinguishes between:

- current implemented behavior
- current package boundaries
- intentional design direction
- future planned semantics

Those are not the same thing.

Whenever Stroid has a real tradeoff, the documentation should state it directly instead of smoothing it over with vague language.

## Current Book Status

This 2.0 documentation set is now organized as the book-format documentation for Stroid.

That means the intended reading model is stable:

- units map to real package surfaces
- chapters map to real runtime concerns
- examples are meant to explain current behavior, not aspirational behavior

Use it as the primary long-form documentation reference for the project.
