# Preface

Stroid did not start from the idea that the world needed another state library.

It came from a more ordinary frustration:

- too many state tools are small but incomplete
- too many state tools are flexible but vague
- too many state tools are fast at demos but unclear under real application pressure

Stroid takes a different position.

It is opinionated about a few things:

- state should have names
- store behavior should be declared in one place
- optional features should be explicit
- the package story should be honest

This book follows the same rule.

It will not pretend:

- that Stroid is the best choice for every app
- that every feature is equally mature
- that bundle size matters more than clarity
- that developers benefit from marketing language disguised as documentation

What this book tries to do instead:

- explain the real core mental model
- explain the split package design
- show which imports do real work
- separate stable design from future intent
- help a developer make a serious decision, not an emotional one

If a part of Stroid is strong, this documentation should say so clearly.

If a part is still a tradeoff, this documentation should say that just as clearly.

That is the standard for the rest of this book.
