# Colophon

This documentation set is written as a repository-native technical book.

That means the design logic is different from a normal polished publisher workflow.

## Current Build Characteristics

- format: Markdown
- location: in-repo documentation under `docs_2.0`
- role: technical documentation book, not print-first layout
- status: current maintained documentation edition

## Writing Principles Used in This Book

This book intentionally prioritizes:

- technical honesty over polish theater
- architectural clarity over generic marketing language
- explicit boundaries over vague feature talk
- pattern consistency across chapters
- direct statements about tradeoffs

## Structural Principle

Each unit is written in a repeatable instructional pattern:

- unit opener
- chapter opener
- learning objectives
- chapter outline
- numbered sections
- examples, tables, figures, and case studies
- summary
- review questions
- exercises
- references

This is not only a stylistic choice.

It reduces reader disorientation and makes the documentation feel dependable. A reader should not have to relearn how to read the book every chapter.

## Tooling Reality

This documentation was developed in the same workspace as the codebase.

That matters because the writing is grounded in:

- the current source
- the current tests
- the current package boundaries
- current known tradeoffs

It is not written from a separate marketing brief.

## Edition Limitation

This colophon would be different in a formal print edition.

A later publication-stage version would likely include:

- final typography decisions
- page design notes
- release/version print history
- formal production credits

For now, the honest colophon is simple:

this is a technical documentation book shaped alongside the system it describes.
